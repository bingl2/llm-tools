# /timebox - 타임박싱 하루 플랜

타임박싱 방식으로 하루를 설계합니다. Big 3(오늘의 핵심 3가지)를 정하고, 시간 블록으로 배치합니다.

## 경로
`$TIMEBOX_HOME` (미설정 시 `~/timebox`). 없는 하위 디렉토리는 자동 생성.

## 핵심 철학

- **Task 중심이 아닌 Time 중심**: "무엇을 하나"가 아니라 "언제 무엇을 하나"
- **Big 3**: 하루에 핵심 3가지만 정하고 나머지는 유연하게
- **에너지 관리**: 고에너지 시간에 Deep Work, 저에너지 시간에 Shallow Work
- **정기 체크인**: 블록 전환 시 짧은 리플렉션으로 하루를 조정 (주기는 `_config.md`의 `checkin_interval`)

## 실행 흐름

### 1단계: 날짜 및 컨텍스트 확인

1. 오늘 날짜와 요일 확인
2. `{base}/goals/` 에서 목표 파일 확인:
   - `{올해}-foundation.md` Read (있으면) — 방향, 제약 로드. 자기 지식 섹션이 있으면 함께 로드
   - `_config.md` Read (있으면) — 체크인 주기, 블록 길이 등 운영 설정 로드
   - `{올해}.md` (연간 목표) Read (있으면)
   - `{올해-이번달}.md` (월간 목표) Read (있으면)
   - `{올해}-W{이번주}.md` (주간 목표) Read (있으면)
3. `{base}/plans/` 에서 가장 최근 파일을 Glob으로 확인
   - 있으면 Read하여 어제의 미완료 Big 3, 에너지 패턴 참고
4. `{base}/reviews/` 에서 가장 최근 review 파일 확인
   - 있으면 Tomorrow Candidates 참조
5. Google Calendar MCP가 연동되어 있으면(`_config.md`의 `google_calendar: on`) 오늘 일정 조회하여 고정 블록으로 예약

**goals 파일이 없으면:**
- "목표 파일이 아직 없습니다. `/timebox-setup`으로 시스템 설정 후 `/timebox-align`으로 목표를 설정하시겠습니까?"
- 사용자가 스킵하면 goals 없이 진행 (graceful skip)

### 2단계: 마스터 파일 확인

`{base}/plans/{오늘날짜}.md` 파일 존재 여부 확인

**파일이 이미 존재하면**:
- 현재 내용을 Read하여 사용자에게 보여줍니다
- AskUserQuestion: "오늘 타임박스가 이미 있습니다. 어떻게 하시겠습니까?"
  - 블록 추가/수정
  - 처음부터 다시
  - 현재 상태 확인만

**파일이 없으면**: 3단계로 진행

### 3단계: 목표 정렬 + Big 3 코칭 대화

**Step A: 목표 컨텍스트 로드** — Claude가 읽고 요약

1. goals/ 파일에서 로드한 내용을 요약
2. 주간 목표가 있으면 이번 주 해당 섹션 요약
3. 어제 review가 있으면 Tomorrow Candidates 참조
4. carry-over (어제 미완료 Big 3) 확인

**목표가 없는 경우 (graceful skip):**
- 목표 파일 없이도 Big 3 설정 가능
- "목표 파일이 없어 프리스타일로 진행합니다. 나중에 `/timebox-align`으로 목표를 세우면 더 강력해집니다."

Claude가 먼저 정리해서 보여줌:
```
"이번 달 목표는 [1, 2, 3]이고, 이번 주 focus는 [X]입니다.
어제 미완료: [Y]. 이 맥락에서 오늘을 설계해봅시다."
```

**Step B: 브레인덤프**

- "목표를 봤으니, 오늘 머릿속에 있는 것들을 전부 쏟아내주세요."
- 사용자가 자유롭게 나열 (5개든 10개든)
- Google Calendar 일정이 있으면 함께 제시

**Step C: 분류 + 목표 연결** — Claude가 분류

- **Outcome (결과)**: "~를 완료/달성한다" → Big 3 후보
- **Task (작업)**: "~를 한다" → 서브태스크 후보
- **Maintenance (유지)**: 이메일, 리뷰, 슬랙 → Shallow 블록으로
- 각 항목에 목표 연결 태그 부여:
  `[월간#2] API 리팩토링 완료` / `[연결 없음] 슬랙 정리`

**Step D: 아웃컴 검증** — 사용자의 시간의 수호자

**이 시점에서 Claude는 사용자의 시간을 지키는 역할이다.** 사용자의 시간은 유한하고 비싸다. 두리뭉술한 목표에 하루를 쓰는 건 그 시간을 버리는 것과 같다. 악착같이 구체화하라.

**필수 검증 — Big 3 후보 중 모호하거나 구체성이 부족한 항목에 적용 (이미 명확한 항목은 스킵):**

1. **Outcome 검증**: "이걸 끝냈을 때, 세상에 뭐가 달라져 있어요?"
   - 답이 모호하면: "그건 활동이지 결과가 아닙니다. 결과를 말해주세요."
   - Task 수준("~를 한다")이면: "이건 작업이지 결과가 아닙니다. 이걸 통해 달성하려는 Outcome은?"

2. **완료 기준 검증**: "'됐다'를 30초 안에 확인할 수 있어야 합니다. 어떻게 확인하죠?"
   - "PR 올린 상태" ✓ / "코드 개선" ✗ / "검토 완료" ✗ (뭘 검토? 어디까지?)
   - 한 문장으로 못 말하면 스코프가 너무 큰 것이다.

3. **시간 대비 가치**: "이거 {n}시간 쓸 가치가 있어요? 이 시간에 할 수 있는 가장 임팩트 있는 일이 정말 이건가요?"
   - 주간/월간 목표 대비 ROI 강제 비교
   - "이번 주 focus는 [X]인데, 이게 거기에 기여하나요?"
   - "월간 목표 [#N]의 진행률이 낮은데, 그쪽을 먼저 해야 하지 않나요?"
   - 목표와 무관한 항목: "이건 월간/주간 어디에도 연결 안 됩니다. 정말 오늘의 Big 3에 넣을 만큼 중요한가요?"

4. **구체성 검증**: "내일 다른 사람이 이 Big 3를 보고 바로 시작할 수 있어요?"
   - 못 한다면 스코프가 불명확한 것. 쪼개거나 구체화 요구.

5. **회피 탐지**:
   - 3일+ carry-over: "3일째 미루고 있습니다. 솔직히 — 이거 하기 싫은 건가요, 못 하는 건가요? 하기 싫으면 드롭하고, 못 하는 거면 왜 못 하는지 같이 풀어봅시다."
   - 쉬운 일만 선택: "어려운 걸 빼고 쉬운 것만 골랐네요. 오늘 하루를 투자할 가치가 있는 조합인가요?"
   - "이게 오늘 안 되면 이번 주 목표가 위험해지나요? 위험하지도 않은 일에 하루를 쓰려는 건 아닌지 확인하고 싶어요."

**톤 원칙:**
- 냉정하되 적대적이지 않게. 목적은 공격이 아니라 사용자가 자기 시간의 가치를 자각하게 하는 것.
- 사용자가 명확하게 답하면 즉시 인정하고 넘어간다. "좋습니다. 명확합니다."
- 끝까지 모호하면: "이 상태로 블록에 넣으면 하루 끝에 '뭘 했지?' 할 가능성이 높습니다. 한 번만 더 구체화해볼까요?"

**Step E: 확정 + 현실 체크**

- Big 3 확정 후 가용 Deep Work 블록 수와 비교
- "Deep Work {n}슬롯({n}시간)에 이 3개가 들어갈까요?"
- 안 들어가면: "3개 중 하나를 내일로 미루거나, 범위를 줄이세요."
- 각 Big 3의 **완료 기준**을 명시적으로 합의: "이게 되면 완료인 거죠?"
- 최종 확인: "오늘 이 3개를 해내면, [주간 focus]에 확실히 기여하는 하루입니다. 맞죠?"

**추정 정확도 보정** (`{올해}-foundation.md`의 자기 지식에 `추정 정확도` 데이터가 있을 때):
- 사용자의 블록 추정에 과거 정확도를 대조하여 안내:
  - "지난 {n}일간 블록 추정 정확도: 평균 {n}배 초과. {작업 유형}은 특히 {n}배 더 걸렸습니다."
  - "이 추정에 같은 배율 적용하면, 합계 {n}블록 → 실제 {n}블록 예상. 가용 {n}블록에 안 들어갑니다."
- 보정 제안: "1블록이 아니라 2블록으로 잡거나, 범위를 줄이는 걸 추천합니다."
- 추정 정확도 데이터가 없으면 이 단계를 스킵한다 (데이터 없이 추측 금지).
- (추정 정확도는 `/timebox-review` 실행 시 Foundation에 축적됩니다)

**Big 3 스코핑 가이드** (Step E에서 반드시 체크):

1. **시간을 고려한 스코핑**: Big 3는 "하고 싶은 것"이 아니라 "오늘 블록 안에 끝낼 수 있는 것"이다. "코드 리뷰 + 개선"처럼 뭉뚱그리지 말고, "트랜잭션 분리까지가 오늘 완료"처럼 **블록 수에 맞는 범위**로 잘라야 한다.
2. **각 Big 3에 예상 블록 수 명시**: Big 3 옆에 `(~1블록)`, `(~2블록)` 표기. 합계가 가용 Deep Work 블록을 초과하면 즉시 조정.
3. **Big 1도 OK**: 3개를 억지로 채우지 않는다. "적게, 깊게"가 원칙. 다만 큰 덩어리 1개라면 **Outcome 기준으로 쪼개는 것**을 제안:
   - 예: "API 리팩토링" → Big 1: 트랜잭션 분리 완료 / Big 2: 경로 수정 완료 / Big 3: PR 올릴 수 있는 상태
   - 핵심: 각각이 독립적으로 "됐다/안됐다" 판단 가능한 Outcome이어야 함
   - 이렇게 나누면 "절반 했다" 대신 "2/3 달성"이라는 구체적 성취감을 얻는다
4. **완료 기준이 모호하면 다시 질문**: "이게 되면 완료"를 한 문장으로 못 말하면, 스코프가 너무 큰 것이다.

### 4단계: 시간 블록 배치

**블록 유연성:**
- 기본 템플릿은 제안일 뿐, 사용자 근무 패턴에 맞게 조정
- {올해}-foundation.md의 제약 사항(일일 Deep Work 블록 수 등) 반영
- "오전에 보통 몇 시간 집중할 수 있나요?" 질문으로 블록 크기 커스텀
- 미팅이 많은 날: 블록을 짧게 (60분 Deep + 15분 Break)
- 집중 가능한 날: 블록을 길게 (120분 Deep)
- 핵심은 블록 크기가 아니라 "이 시간에는 이것만 한다"는 약속

1. 기본 템플릿에 Big 3를 배치합니다:
   - 오전 Deep Work (고에너지) → Big 3 중 가장 어려운 것
   - 오후 초반 Shallow → 미팅, 이메일, 리뷰
   - 오후 Deep Work → Big 3 나머지
   - 마지막 Flex → 미완료 또는 ad-hoc
2. Google Calendar 일정이 있으면 해당 시간대를 고정 블록으로 배치
3. AskUserQuestion으로 배치 안을 보여주고 확인/수정:
   - "이렇게 배치했습니다. 조정할 부분이 있나요?"

### 5단계: 마스터 파일 생성

Write 도구로 파일을 생성합니다.

**경로**: `{base}/plans/{YYYY-MM-DD}.md`

```markdown
# {YYYY-MM-DD} ({요일}) Timebox

## Big 3
1. [ ] {핵심 과제 1} (~{n}블록)
2. [ ] {핵심 과제 2} (~{n}블록)
3. [ ] {핵심 과제 3} (~{n}블록)

## Schedule

### 09:00-10:30 | Deep Work
**Focus**: {Big 3 #1}
- [ ] {구체적 작업}
- [ ] {구체적 작업}

### 10:30-10:45 | Break
> 체크인: 집중도? 다음 블록 조정?

### 10:45-12:00 | Deep Work
**Focus**: {Big 3 #2 또는 #1 계속}
- [ ] {구체적 작업}

### 12:00-13:00 | Lunch

### 13:00-14:00 | Shallow Work
- [ ] 이메일/슬랙 처리
- [ ] 코드리뷰
- [ ] {기타 경량 작업}

### 14:00-15:30 | Deep Work
**Focus**: {Big 3 #3}
- [ ] {구체적 작업}

### 15:30-15:45 | Break
> 체크인: 오후 에너지? 남은 블록 계획?

### 15:45-17:00 | Flex
> 미완료 블록 이어하기 또는 ad-hoc 처리

### 17:00-17:30 | Wrap-up
> `/timebox-end`로 마감

## Energy Log
| Time | Energy | Notes |
|------|--------|-------|

## Notes

```

**블록 타입 설명**:
- **Deep Work**: 집중 블록. Big 3 작업용. 슬랙/메일 끄고 몰입.
- **Shallow Work**: 이메일, 리뷰, 미팅 등 경량 작업.
- **Break**: 전환 + 체크인. **종료 의식** 포함 (아래 참조).
- **Flex**: 미완료 캐치업 또는 예상 못한 작업.
- **Wrap-up**: 하루 마감 정리.

**블록 배치 원칙 — 항아리와 돌**:
- Big 3 블록을 **가장 먼저** 캘린더에 배치한다 (큰 돌 먼저)
- 그 다음 Shallow/Maintenance를 빈 틈에 채운다 (모래)
- Big 3 블록이 다 들어가지 않으면, Big 3를 줄이거나 범위를 축소한다
- 절대로 Shallow를 먼저 배치하고 남는 시간에 Big 3를 끼워넣지 않는다

**종료 의식 (Block Exit Ritual)**:
- Break 블록에 진입하면 반드시 **물리적 마침표**를 안내:
  - "자리에서 일어나세요. 물 한 잔, 기지개, 창밖 보기."
  - "방금 블록의 맥락을 머릿속에서 비우고, 다음 블록을 준비하세요."
- 목적: "조금만 더 하면 되는데..."를 끊어내는 장치

**블록 내부 쪼개기 (Deep Work 120분 이상일 때)**:
- 120분 이상의 긴 Deep Work 블록은 내부에 미니 휴식 제안:
  - 50분 집중 + 10분 휴식, 또는 25분 + 5분 (뽀모도로)
  - "2시간 블록입니다. 중간에 한 번 끊을까요? (50+10 / 25+5 / 안 끊음)"
- 90분 이하 블록은 쪼개지 않음 (한 호흡으로 충분)

**버퍼 타임**:
- 블록과 블록 사이에 5~10분 여유를 둔다
- Break 블록이 이 역할을 겸하지만, Break 없이 블록이 이어질 때는 명시적 버퍼 추가
- 앞 블록이 밀려도 다음 블록 시작이 망가지지 않도록 하는 완충 장치

### 6단계: 체크인 루프 자동 시작 & 완료 안내

1. 생성된 파일 경로와 Big 3 요약을 안내합니다
2. 현재 시각 기준 "지금은 [{블록명}] 시간입니다. {Focus} 시작하세요!" 안내
3. "`/timebox-now`로 다음 블록을 확인하세요." 안내
4. **체크인 루프 설정**:
   - CronList로 기존 `/timebox-loop` cron 확인
   - 이미 있으면: "체크인이 {n}분 주기로 돌고 있습니다. 변경하시겠습니까?" (변경 시 기존 삭제 → 재생성)
   - 없으면: "체크인 루프를 시작할까요? (기본: {_config.md의 checkin_interval 또는 15}분)"
   - AskUserQuestion으로 응답 → `/loop {n}m /timebox-loop` 실행 안내
   - 이 루프는 `/timebox-end` 실행 시 해제됨

## 참고사항

- 마스터 파일 경로: `{base}/plans/{YYYY-MM-DD}.md`
- 목표 파일 경로: `{base}/goals/`
- 블록 기본 길이: Deep Work 90분, Shallow 60분, Break 15분 (`_config.md`에서 변경 가능)
- **1블록 = `_config.md`의 `deep_work_block` 설정 시간** (기본 90분). 블록 길이가 날마다 달라도 Estimation Accuracy의 배율(예상 대비 실제)은 비교 가능
- 사용자 근무시간에 맞게 블록을 조정 가능
- Big 3는 최대 3개. "적게, 깊게"가 원칙
